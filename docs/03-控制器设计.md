## 3.1 控制器职责

`reverse-controller.py` 是整个系统的 Brain，承担以下职责：

1. **理解** — 解析任务意图，确定是 Web 还是 APP 逆向
2. **拆解** — 将任务拆成有向无环图 (DAG) 的步骤
3. **决策** — 每个步骤完成后，判断下一步：继续 / blocked / skip / done
4. **分配** — 根据步骤类型选择执行器 (Codex / M2.7 / 专用工具)
5. **归档** — 结果写入 memory 层，更新 SESSION-STATE.md

## 3.2 Web 逆向控制器流程

```
输入: WebTask (url, method, actions, known_params...)

Step 1: 信息收集 (recon)
        └── 工具: browser-control.py
        └── 输出: target-info.json
            { url, method, headers, cookies, api_calls[], js_files[] }

Step 2: 抓包分析 (capture)
        └── 工具: mitmproxy
        └── 输出: capture.har
        └── 分支:
            ├── 有明文 sig → 直接进 Step 4
            └── sig 加密/混淆 → 进入 Step 3

Step 3: 浏览器探测 (browser_probe)
        └── 工具: Chrome Headless + stealth scripts
        └── 输出: browser-fingerprint.json
            { ua, tls_fingerprint, js_fingerprint }

Step 4: JS 定位 (js_locate)
        └── 工具: browser-control.py (hook console.debug)
        └── 输出: js-entry-points.json
        └── 分支:
            ├── 直接定位 → 进入 Step 5
            └── 需要反混淆 → 先 Step 4b 再 Step 5

Step 4b: 混淆还原 (deobfuscate)
        └── 工具: js-deobfuscator
        └── 输出: deobfuscated.js

Step 5: 逆向分析 (reverse)
        └── 工具: Codex (主推) / M2.7 (二次验证)
        └── 输出: signature-analysis.md
            { algorithm, key?, iv?, padding?, code_snippet }

Step 6: 脚本生成 (implement)
        └── 工具: Codex / M2.7
        └── 输出: <target>-signature.py

Step 7: 验证 (verify)
        └── 工具: verify-signature.py
        └── 输出: verify-result.json
            { matches: bool, avg_diff_ms, test_count }

Step 8: 二验 (second_opinion)
        └── 工具: M2.7
        └── 输出: review-report.md

Step 9: 归档 (archive)
        └── 自动写入 memory 层
        └── 输出: archive-complete
```

## 3.2.1 Web 逆向 — 方案一：Browser Hook 分支

> **适用场景**：JS 算法极其复杂（如重度混淆、反调试、动态加载），但浏览器环境中已暴露 `window.mnsv2` 等签名函数时。
> 优势：绕过逆向分析，直接调用原函数，**成功率接近 100%**。

```
输入: WebTask (url, method, actions, known_params...)

Step 1: 信息收集 (recon)
        └── 工具: browser-control.py
        └── 输出: target-info.json
        └── 分支:
            ├── signIncludesUrl 命中 + window.mnsv2 可用 → 进入 Hook 路径
            └── 其他情况 → 进入标准路径（Step 2）

┌─────────────── HOOK 路径（方案一）─────────────────┐
│                                                    │
│ Step H1: Hook 探测 (hook_detect)                   │
│         工具: xhs-hook-sign.py                     │
│         操作: Playwright 启动 Chrome → 注入 Hook   │
│         验证: window.mnsv2 是否为 function         │
│         输出: hook-detect.json                     │
│             { mnsv2_found: bool,                   │
│               mnsv2_type: "function"|"undefined",  │
│               candidates: [...],                   │
│               sign_patterns: [...] }                │
│         分支:                                      │
│           ├── mnsv2 找到 → 进入 H2                 │
│           └── 未找到 → 回退标准路径 Step 5         │
│                                                    │
│ Step H2: Hook 生成签名 (hook_sign)                  │
│         工具: xhs-hook-sign.py                     │
│         操作: page.evaluate(mnsv2(url, body, {}))  │
│         输出: hook-sign-result.json               │
│             { x_s: "...", x_t: "...", headers }   │
│                                                    │
│ Step H3: 脚本生成 (implement_hook)                 │
│         工具: xhs-hook-sign.py generate-script    │
│         输出: <target>-signature.py               │
│             → 封装 Playwright + mnsv2 调用         │
│             → 对外暴露 XHSSigner.sign(path, body) │
│                                                    │
│ Step H4: 验证 (verify)                            │
│         工具: verify-signature.py                 │
│         输出: verify-result.json                  │
│                                                    │
│ Step H5: 归档 (archive)                          │
└────────────────────────────────────────────────────┘
```

### 方案对比

| | 标准路径（方案二） | Hook 路径（方案一） |
|---|---|---|
| Step 5 | 逆向 JS 算法（~30分钟） | ❌ 跳过 |
| Step 6 | Python 重写算法（~10分钟） | 生成 Playwright 包装脚本 |
| 成功率 | 60-80%（依赖算法复杂度） | **95%+** |
| 速度 | 签名生成快（<1ms） | 签名生成慢（~1-3秒） |
| 依赖 | Python 环境 | 浏览器 + Playwright |
| 适用 | 高频调用、长期运营 | 低频、复杂算法、临时任务 |

### Hook 路径判断条件

在 `Step 1 (recon)` 完成后，控制器自动判断进入哪条路径：

```python
def choose_pipeline(hook_detect_result: dict) -> str:
    """判断使用哪条 Pipeline"""
    if hook_detect_result.get("mnsv2_found") is True:
        return "hook"        # 方案一：直接 Hook
    elif hook_detect_result.get("algorithm_complexity") == "high":
        return "hook"        # 算法复杂，优先 Hook
    else:
        return "standard"    # 方案二：标准逆向
```

> 💡 **实战建议**：先用 Hook 路径快速出结果交付，再切标准路径做纯算法版本（高频场景需要）。两套可以并存。

## 3.3 APP 逆向控制器流程

```
输入: AppTask (package, apk_path, target_api, hook_targets)

Step 1: APK 获取 + 基础分析 (apk_fetch)
        └── 工具: APKTool / JADX
        └── 输出: apk-info.json
            { package, version, activities, permissions, sdk_version }

Step 2: mitmproxy 抓包 (mitm_capture)
        └── 工具: mitmproxy (+ SSL Pinning 绕过)
        └── 输出: app-capture.har
        └── 输出: target-requests.json
            { api_endpoint, request_body, response_body, sign_param }

Step 3: Frida 动态 Hook (frida_hook)
        └── 工具: frida-task-executor.sh
        └── 输出: frida-log.txt
        └── 输出: hooked-methods.json
        └── 分支:
            ├── Java 层搞定 → 进入 Step 5
            └── 需要 SO 层 → 先 Step 4

Step 4: SO 层分析 (so_analyze)
        └── 工具: IDA Pro / Ghidra / Frida
        └── 输出: so-analysis.md
            { function_name, algorithm, registers, pseudo_code }

Step 5: Python 实现 (implement)
        └── 工具: Codex / M2.7
        └── 输出: <package>-signature.py

Step 6: 验证 (verify)
        └── 工具: verify-app-signature.py
        └── 输出: verify-result.json

Step 7: 归档 (archive)
```

## 3.4 控制器决策伪代码

```python
class ReverseController:
    def __init__(self, task):
        self.task = task
        self.pipeline = self.build_pipeline()
        self.state = {}

    def run(self):
        for step in self.pipeline:
            # 检查是否 blocked
            if self.is_blocked(step):
                self.update_session_state(blocker=step.blocker_reason)
                break

            # 检查是否 skip
            if self.should_skip(step):
                step.status = "skipped"
                continue

            # claim 并执行
            job = self.create_job(step)
            result = self.execute_job(job)

            # 判断结果
            if result.verified:
                step.status = "verified"
                self.archive(result)
            elif result.failed and step.can_retry:
                step.status = "retry"
                self.retry(step)
            else:
                step.status = "failed"
                self.update_session_state(blocker=step.blocker_reason)
                break

        self.update_session_state(done=(step.status == "verified"))

    def build_pipeline(self):
        if self.task.type.startswith("web."):
            return WebReversePipeline(self.task).steps
        elif self.task.type.startswith("app."):
            return AppReversePipeline(self.task).steps
        else:
            return CommonPipeline(self.task).steps
```

## 3.5 状态机设计

```
                    intake (新建)
                        ↓
              ┌──── analyzing (分析) ────┐
              ↓                          ↓
        waiting_tools              waiting_capture
        (等工具环境)                 (等抓包结果)
              └──────────┬───────────┘
                         ↓
              ┌──── reversing (逆向中) ────┐
              ↓                            ↓
        verified ✅                    failed ❌
        (结果通过验证)                   │
              │                         ↓
              │               ┌──── blocked ⏸ ────┐
              │               ↓                   ↓
              │         can_retry?           no_retry?
              │            ↓                    ↓
              │        retry n              write to
              │        (≤3次)               SESSION-STATE.md
              │            ↓                   ↓
              │      ┌─────────┐        wait_for_human
              └─────→ done ✓ ←┘         or_new_sample
```

## 3.6 Blocker 处理机制

当任务进入 blocked 状态时：

1. 写入 `SESSION-STATE.md`：
   ```
   ## Blocker
   - 项目: xxx
   - 阶段: Step 5 (逆向分析)
   - 原因: SO 层算法在 ARM 汇编中，无法直接还原
   - 阻塞时间: 2026-04-22T14:00:00Z
   - 可能方案:
     1. 尝试 Frida Hook 运行时获取参数
     2. 找相同 SDK 的开源实现
     3. 人工介入分析
   ```

2. 任务进入等待态，不继续空转

3. 定时器每 6 小时扫一次 blocked 任务：
   - 有新样本/新方案 → 自动续跑
   - 无变化 → 持续等待
