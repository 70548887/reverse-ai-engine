## 1.1 设计原则

1. **控制器驱动**：AI Controller 是唯一决策节点，所有执行器都从控制器获取指令
2. **事件替代轮询**：样本落地触发，timer 仅作保底，减少空转
3. **工单判重**：同一 `(stage, fingerprint)` 只能存在一个未完成 job
4. **记忆持久化**：热记忆 → 每日 → 项目 → Raw → History，层层归档
5. **可追溯**：每个结果附带 SHA256，随时可验证"哪个文件、哪次执行"

## 1.3 数据流

```
任务下发
   ↓
写入 tasks/intake/pending/<task_id>.jsonl
   ↓
reverse-controller.py 扫描 intake → 创建工单
   ↓
写入 tasks/queue/<stage>.jobs.jsonl
   ↓
执行器 claim job → 执行 → 写 result.json
   ↓
auto-task-runner.py 镜像到各层：
   • daily memory (当日结论)
   • project memory (专题归档)
   • raw results (完整 result.json)
   • history (轨迹镜像)
   ↓
import-memory-to-lancedb.mjs 导入向量库
   ↓
SESSION-STATE.md 更新当前状态
   ↓
Git 提交审计
```

## 1.4 核心状态文件

| 文件 | 位置 | 用途 |
|------|------|------|
| `SESSION-STATE.md` | `workspace/` | 热记忆：当前主线/blocker/下一步 |
| `pipeline-state.json` | `tasks/queue/` | 当前 pipeline 整体状态 |
| `<stage>.jobs.jsonl` | `tasks/queue/` | 各 stage 工单队列 |
| `*.latest.json` | `tasks/status/` | 各任务最新执行结果 |
| `*.history.jsonl` | `tasks/status/` | 执行历史轨迹 |
| `memory-sync.json` | `tasks/status/` | 记忆同步状态 |

## 1.5 与现有 bilibili-reverse 架构的关系

本架构继承并扩展了 bilibili-reverse 项目已有的成熟机制：

- ✅ 工单判重 (jobs.jsonl)
- ✅ 控制器决策 (bilibili-controller.py)
- ✅ 事件驱动 (systemd path/timer)
- ✅ 双执行器分流 (Codex + M2.7)
- ✅ 五层记忆体系
- ✅ LanceDB 向量召回
- ✅ Git 审计
