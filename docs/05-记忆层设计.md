## 5.1 记忆分层体系

```
┌─────────────────────────────────────────────────────────────────┐
│  Layer 0: 🔥 HOT RAM                                           │
│  文件: SESSION-STATE.md                                         │
│  内容: 当前主线 / blocker / 下一步动作                          │
│  用途: 抗上下文压缩，会话续接 / 长任务续做                      │
│  风险: 多端同时写会冲突，建议单写                               │
├─────────────────────────────────────────────────────────────────┤
│  Layer 1: 📅 Daily Memory                                       │
│  文件: memory/YYYY-MM-DD.md                                     │
│  内容: 当天增量（阶段结论、方法、踩坑、自动任务结果）           │
│  用途: 保存每日工作轨迹                                         │
├─────────────────────────────────────────────────────────────────┤
│  Layer 2: 📁 Project Memory                                      │
│  文件: memory/projects/<project>/                               │
│  内容: 专题长期结论（从 daily 提炼，重复出现的方法/结论）       │
│  用途: 把阶段结论收成"模型"，比 daily 更有复用价值              │
├─────────────────────────────────────────────────────────────────┤
│  Layer 3: 📦 Raw Results Archive                                 │
│  文件: memory/projects/*-auto-results.md                       │
│  内容: 完整 result.json 原文                                     │
│  用途: 防止"只剩摘要，以后又得重挖"                             │
├─────────────────────────────────────────────────────────────────┤
│  Layer 4: 🔀 History Mirror                                      │
│  文件: memory/projects/*-auto-history.md                        │
│  内容: 状态轨迹镜像（tasks/status/*.history.jsonl 的关键 run）  │
│  用途: 避免"状态轨迹只留在 status 目录"                         │
├─────────────────────────────────────────────────────────────────┤
│  Layer 5: 🔍 Vector (LanceDB)                                    │
│  文件: ~/.openclaw/memory/lancedb/                              │
│  内容: 所有记忆的语义向量，1300+ 条                              │
│  用途: "查得到" — 通过语义相似性召回旧经验                      │
├─────────────────────────────────────────────────────────────────┤
│  Layer 6: 🗂️ Git Audit                                          │
│  目录: memory/ 和 workspace/ 的 git 仓库                       │
│  内容: 所有记忆文件的版本历史                                    │
│  用途: "可追溯" — 任何改动可回滚、可对比                        │
└─────────────────────────────────────────────────────────────────┘
```

## 5.4 项目专题记忆格式

```markdown
# memory/projects/web-reverse/techniques.md

## 已验证的逆向技术

### 过反调试
- **适用**: 大多数国内App WebView + JS混合页面
- **方法**: Puppeteer + stealth.min.js + Object.freeze(devtools)
- **命令**: `python3 browser-control.py --stealth --target <url>`
- **验证状态**: ✅ 已验证

### AES 签名提取
- **适用**: crypto-js AES 加密的 sign 参数
- **方法**: Hook `CryptoJS.AES.encrypt` 捕获 key/iv
- **Frida脚本**: frida-scripts/cryptojs-aes-hook.js
- **验证状态**: ✅ 已验证

### JSfuck 混淆还原
- **适用**: JSfuck / AAEncode 混淆
- **方法**: 直接在 Chrome Console 执行混淆字符串得到明文
- **工具**: browser-control.py 内置 jsfuck-decode
- **验证状态**: ⚠️ 部分验证
```

## 5.6 向量召回

### 导入脚本

```javascript
// tools/import-memory-to-lancedb.mjs

import { lancedb } from "./lancedb-client.mjs";

const records = [];

for (const file of memoryFiles) {
    const content = readFile(file);
    const type = inferType(file); // technique / signature / target / blocker

    records.push({
        id: stableHash(file + content),
        type,
        content,
        tags: extractTags(content),
        file,
        imported_at: new Date().toISOString()
    });
}

await lancedb.insert(records);
console.log(`Imported ${records.length} records to LanceDB`);
```

### 召回健康检查

```python
# scripts/memory-recall-healthcheck.py

FIXED_QUERIES = [
    "AES signature extraction JS逆向",
    "Frida SSL pinning bypass Android",
    "JSfuck obfuscation还原",
    "某网站 sign 参数生成算法",
]

def health_check():
    for query in FIXED_QUERIES:
        results = lancedb.search(query, top_k=3)
        print(f"Query: {query}")
        print(f"  Hits: {len(results)}")
        print(f"  Top match: {results[0]['file'] if results else 'NONE'}")
```

### 召回流程

```
用户问: "某视频站的 sign 参数怎么逆向？"
   ↓
向 LanceDB 搜索: "sign 参数 逆向 视频站"
   ↓
召回相关记忆:
   - memory/projects/web-reverse/target-dy.md (以前逆向过)
   - memory/projects/web-reverse/techniques.md (技术库)
   ↓
把召回结果注入 Controller 上下文
   ↓
Controller 参考旧经验，决定：
   - 直接复用类似脚本
   - 需要新逆向（节省 50% 时间）
```

## 5.7 记忆同步机制

```
auto-task-runner.py 每轮执行后自动同步：

1. 写 result.json → tasks/status/
2. 镜像到 daily memory → memory/YYYY-MM-DD.md
3. 镜像到 project auto log → memory/projects/*-auto-log.md
4. 镜像到 project raw results → memory/projects/*-auto-results.md
5. 镜像到 project history → memory/projects/*-auto-history.md
6. 触发 LanceDB 导入 (systemd timer, 每30分钟)
7. 写 memory-sync.json (记录同步状态)
8. Git 提交 (systemd timer, 每天3点)
```

## 5.8 多端风险控制

| 层级 | 多端风险 | 缓解措施 |
|------|---------|---------|
| SESSION-STATE.md | ⚠️ 高（单文件热写） | 单写原则，只有一个主控端写 |
| daily memory | 🟡 中（追加但可能冲突） | 带 job_id 去重标记 |
| project memory | 🟢 低（通常不同项目） | 按项目目录隔离 |
| raw results | 🟢 低（追加不覆盖） | 只追加新结果文件 |
| LanceDB | 🟡 中（重复导入同一内容） | 用 stable hash 做 id |
| Git | 🟢 低（冲突可见可回滚） | 定期 git pull --rebase |
